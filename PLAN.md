## Phase 1 — Scene-Graph Document Model + GFX/Konva Editor — COMPLETE (2026-07-04)

The architectural foundation: one serializable document that editors mutate and renderers read. Every later phase (Virtual Set, AR, sports package, data bindings, output plane) layers on top of the shape decided here.

**Re-verified live (2026-07-04, this session):** found a stale orphaned `broadcast-engine.exe`/Vite pair from a prior run still bound to ports 4977/1420 with no live window and a null `/document` — killed it and did a clean `bun run tauri dev` relaunch. `tsc --noEmit` and `cargo check` both pass clean (one expected dead-code warning on unused `ndi.rs` stub methods). Confirmed live: `studio.db`'s persisted project (2 scenes, a lower-third with rect + "HOST NAME"/"Correspondent" text) reloaded correctly through `initPersistence()`; `/document` returned the real envelope; `/program` rendered the actual positioned rect+text HTML (not the blank placeholder) — the full editor→store→SQLite→sidecar→OBS pipeline is real, not just compiling.

**Layer lock verified correct:** `GfxEditor.tsx` computes `interactive = layer.visible && !layer.locked` per layer and passes that into `renderElement`, which sets `listening: false` for the whole layer when locked — a locked layer's elements genuinely cannot be selected or dragged, matching the acceptance test. (`Element.locked` is a separate, still-unused-in-UI per-element field — no Layers/Inspector control sets it yet, so it's inert, not a bug.)

**Acceptance test (written first):**
- **Document model:** fresh launch creates a default 1920×1080/50fps Project with one empty Scene; adding a `gfx2d` layer with 1 rect + 2 text elements gives each a stable id addressable by path (`scenes[0].layers[1].props.elements[2]`).
- **Editor:** selecting an element shows Konva `Transformer` handles; drag/resize/rotate all write back to the store; Inspector fields are bidirectional with the canvas; Layers panel tree shows visibility/lock and reorder; a locked layer can't be selected/dragged.
- **Undo/redo:** one completed drag/resize/rotate = exactly one undo entry (not one per animation frame); panning/zooming/selecting creates **no** undo entry.
- **Persistence:** edits autosave (debounced ~700ms) to `sqlite:studio.db`; close and relaunch restores the identical project/scene/layer/elements at the same geometry and style; a schema-version envelope prevents corruption across restarts.
- **Two-consumer proof:** a pure `renderElement()` builder is used by both the interactive Konva editor and a non-interactive `DocumentRenderer`; the Program window renders the same lower-third at correct 1:1 pixel geometry with a transparent background, no editor chrome.
- **OBS (scope-bounded):** the axum sidecar's `/program` route is upgraded to render the current document's lower-third (static snapshot, not yet animated/live-pushed) instead of the Phase 0 placeholder page; an OBS Browser Source at the same URL shows it. Live/animated push, PGM/PVW, GSAP timelines, and real data bindings are explicitly deferred to Phases 2/3.

**Document model (`src/document/types.ts`):**
```
Project { id, name, resolution, fps, colorSpace, scenes[], assets[], schemaVersion }
Scene   { id, name, layers[] }
Layer   { id, name, kind: 'gfx2d'|'set3d'|'map'|'chart', zIndex, visible, locked,
          opacity, blendMode, transform, bindings[], props: LayerProps }
LayerProps = discriminated union keyed by kind — only 'gfx2d' has real fields in
             Phase 1 ({ elements: Element[], safeArea? }); set3d/map/chart are
             reserved empty variants so later phases add fields with zero migration.
Element = RectElement | TextElement | ImageElement | GroupElement
Binding { targetPath, source, format?, fallback? }   // shape only — engine is Phase 3
Asset   { id, kind: 'image'|'font', name, src }
```
No fixed template menu anywhere in this shape — elements are freely placed and composed. Build via factory functions (`src/document/factory.ts`) only, so `Layer.kind` and `Layer.props.kind` can't drift out of sync.

**Undo/redo:** two slices in one zustand store (`src/document/store.ts`) — `project` (undoable via zundo's `partialize`) vs UI state (selection, active scene/layer, canvas zoom/pan, `dirty` — excluded from history). One-undo-per-gesture via `temporal().pause()`/`.resume()` around drag/transform start/end (`src/document/gestureHistory.ts`).

**Persistence:** JSON-blob-per-project in SQLite, not normalized tables — the document is already a serializable tree, autosave is one atomic `UPDATE`, versioning lives in a `schema_version` column driving a JS-side up-migrator. Migration v2 in `src-tauri/src/lib.rs` (append-only, never edit Phase 0's `_health` migration): `projects(id, name, doc, schema_version, updated_at)`, `app_state(k, v)`. Only the Control Room window touches SQLite (capability-scoped since Phase 0) — this is why Program routes through the sidecar instead of direct DB access.

**Renderer/editor split:** `src/components/gfx/renderNodes.tsx` — pure `Element` → react-konva node mapping, no store/hooks. `GfxEditor.tsx` wraps it in an interactive Stage + Transformer + safe-area overlay (editor-only chrome). `DocumentRenderer.tsx` wraps it in a non-interactive Stage (`listening={false}`, 1:1 resolution) used by both Program and, via a new `GET /document` axum route + `set_program_document` Tauri command, OBS. Polling (~1s) keeps Program live-ish without inventing a bespoke WS protocol ahead of Phase 7's real control server.

**Panels:** GFX Editor, Layers (tree, visibility/lock, reorder, add-element palette, "bind to data" stub), Inspector (bidirectional with canvas) — replacing the Phase 0 Dockview placeholders.

**Files:** new `src/document/{types,schema,ids,factory,store,persistence,gestureHistory}.ts`, `src/components/gfx/{renderNodes,GfxEditor,DocumentRenderer,SafeAreas}.tsx`, `src/components/panels/{GfxEditorPanel,LayersPanel,InspectorPanel}.tsx`; replace `src/state/appStore.ts`; edit `ControlRoomView.tsx`, `ProgramView.tsx`, `src-tauri/src/lib.rs`. New deps: `konva`, `react-konva` only.

Fix used: enumerate processes via `Get-CimInstance Win32_Process` filtered on cmdline containing the app's data-dir name (`broadcast`) and stop all of them, not just the main exe, before relaunching. Worth remembering for any future "verify live" restart in this project.

## Phase 5 — Virtual Set — IN PROGRESS (2026-07-05)

Port `legacy/src/components/studio/VirtualStudioScene.tsx`'s 6 set geometries and 3-point lighting rig into document-driven, gizmo-editable scenes (as `set3d` layers, already reserved in Phase 1). Real `@react-three/postprocessing` (the old `DepthPanel.tsx` was 100% cosmetic — new build, not a port). This is the standalone 3D broadcast environment engine: no Unreal/Unity/Godot dependency, targeted at small machines (quality knobs down to 0.5× DPR, shadows/bloom off by default) with broadcast-grade output.

**Acceptance test (written first):**
- **Document model:** `set3d` LayerProps carries a real node graph — `SetNode = model | primitive | text3d | light | camera | videofeed | group` — each with a full 3D transform (position/rotation/scale), plus per-layer environment (background, floor, grid, ambient, fog) and render settings (exposure, shadows, DPR, bloom, vignette). All of it round-trips through the Zod schema without a field being stripped (the known Phase 4 bug class), verified by a script that parses every set builder's output through `projectSchema`.

**Virtual camera motion engine (`src/document/cameraMoves.ts` + rigs in `Set3dRenderer.tsx`) — smooth takes, push/pull/slide, orbit/follow/focus, and PVW camera rehearsal.** State model deliberately copies `layerPlayback`'s proven contract: a transient, never-persisted slice (`cameraMoves`/`cameraOrbits`/`cameraPreview`, keyed by set3d layer id) pushed in the program envelope with a shared `startedAt` wall-clock timestamp, so Program, Preview, and the embedded monitors each independently reconstruct the identical mid-flight pose with no sync protocol — and the END state is always committed to the document first (`activeCameraId` on take, the camera node's transform on push/slide/focus/orbit-stop), so a consumer that misses the whole animation still lands on the correct final frame, and every automated move is one clean undo entry. Render-side, a transient rig camera (drei `PerspectiveCamera makeDefault`) mounts AFTER the node graph while a move/orbit is live — its makeDefault wins, unmounting restores the committed node camera — and reads the *destination's live world pose* off the scene graph by object name (`camnode-<id>`/`setnode-<id>`, names added in SetNodes.tsx), so landings are exact even for cameras nested in transformed groups.
